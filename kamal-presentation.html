About the Author

Sam Ruby

  • Primary author of "Agile Web Development with Rails" since 2008
  • Contributor to Ruby on Rails framework
  • Works at Fly.io (PaaS company)
  • Believes in both PaaS AND Kamal - they complement each other

Sam Ruby brings a unique perspective - he works for a PaaS company but advocates for Kamal because he believes in giving developers choices. His dance showcase app runs on 12 hosts serving 326 events in 70 cities across 4 continents.

Just imagine...

What if you could give every user of your software the experience of having a dedicated server machine assigned to them?

This is Sam's vision from his Shared Nothing presentation - the ultimate user experience where everyone feels like they have their own dedicated infrastructure. This is what we're building toward with Kamal.

The Core Message

"Production is more than copying files and starting a process"

Production requires:

  • Database backups
  • Searchable logs
  • Application monitoring
  • Security measures
  • And much more...
This is the key takeaway - many developers think deployment is just getting code on a server, but production readiness requires comprehensive planning.

What is Kamal?

Zero-downtime deployment tool for containerized applications

Features:

  • Rolling restarts
  • Asset bridging
  • Remote builds
  • Accessory service management

Key Points:

  • Built on Docker
  • Originally for Rails
  • Works with any containerized app
Kamal is a deployment orchestration tool that leverages Docker. It handles the complex choreography of deploying applications without downtime. While built for Rails, it's framework-agnostic.

Prerequisites & Preparation

Before You Deploy, Gather:

  • ✅ SSH Key (not passwords!)
  • ✅ Server (VPS or dedicated)
  • ✅ Domain name
  • ✅ Container registry account
  • ✅ Password manager
  • ✅ Optional: Separate builder

Deploy Locally First!

Preparation is crucial. The book emphasizes deploying locally first to catch issues early. Using SSH keys instead of passwords is non-negotiable for security.

Show actual config snippets if possible. Emphasize that most configuration is set-and-forget. The book provides detailed line-by-line explanations.

First Deploy Commands

# Initial setup (installs Docker, builds, deploys)
bin/kamal setup

# Subsequent deploys
bin/kamal deploy

What happens during setup:

  • Installs Docker on host
  • Builds & pushes image
  • Starts container
  • Runs migrations
  • Configures SSL
  • Sets up proxy
The setup command does heavy lifting - it's essentially bootstrapping your entire production environment. Subsequent deploys are much simpler.

Security Best Practices

Three Attack Vectors to Protect:

  1. You - Use SSH keys & password managers
  2. Your Application - Keep dependencies updated
  3. Your Server - Firewall & system updates

Essential Ports:

  • 22 (SSH)
  • 80 (HTTP)
  • 443 (HTTPS)
  • Block everything else!
Security isn't optional. The book emphasizes "trust but verify" - even with proper setup, you should test your security configuration.

Litestream: Continuous SQLite Backups

dbs:
  - path: /data/production.sqlite3
    replicas:
      - type: s3
        endpoint: ${ENDPOINT_URL}
        bucket: ${BUCKET_NAME}
        path: depot
  • Real-time S3 replication
  • Point-in-time recovery
  • Minimal performance impact
Litestream is perfect for SQLite databases - it provides enterprise-grade backup capabilities without complexity. The book also covers PostgreSQL scenarios.

Benefits:

  • Easy configuration

Also covered:

  • Sentry for error tracking

Vector is a hidden gem - it's a Swiss Army knife for observability. The book provides complete configuration examples.

Performance with Thruster

HTTP/2 and Asset Caching

# Simple addition to config.ru
require "thruster"
run Thruster.run(Rails.application)

Benefits:

  • X-Sendfile support
  • HTTP/2 out of the box
  • Automatic asset caching
  • Minimal configuration
Thruster is a simple addition that provides significant performance benefits. It's especially useful for serving static assets efficiently.

Staging Environments

Test before production!

Options covered:

  1. Separate staging server
  2. Multiple apps on one server (using Kamal proxy)
  3. Environment-specific configurations

Key insight: Kamal's proxy allows multiple apps per server

The book shows how to run staging and production on the same server if needed, using Kamal's proxy to route by domain.

Key Takeaways

  1. Deploy locally first - Catch issues early
  2. Gather ingredients before cooking - Preparation is key
  3. Security is not optional - SSH keys, firewalls, updates
  4. Backups are essential - Automate with Litestream
  5. Monitor everything - Logs, errors, performance
  6. Start simple, scale later - Don't over-engineer
These six points summarize the entire book. If attendees remember nothing else, these are the crucial lessons.

When to Use What?

Use Kamal when:

  • You want full control
  • Cost optimization is important
  • You enjoy managing infrastructure
  • You have specific compliance needs

Use PaaS when:

  • You want to focus only on code
  • You need global presence quickly
  • You have a small team
  • You value convenience over cost

Both are valid choices!

The author's unique position allows honest comparison. There's no "right" answer - it depends on your team and requirements.
